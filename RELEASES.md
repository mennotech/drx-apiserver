# Releases

This document describes how `drx-apiserver` artifacts are versioned,
published, and supported. For runtime release notes, see
[base/CHANGELOG.md](base/CHANGELOG.md). For repository-level changes
(CI, docs, tooling), see [CHANGELOG.md](CHANGELOG.md).

---

## Published artifacts

| Artifact            | Source           | Registry / Location                                   | Release track |
| ------------------- | ---------------- | ----------------------------------------------------- | ------------- |
| `drx-drupal-base`   | [base/](base/)   | `ghcr.io/mennotech/drx-drupal-base`                   | Primary, versioned. |
| `drx-apiserver` (reference overlay) | [server/](server/) | `ghcr.io/mennotech/drx-apiserver` *(separate publishing pipeline; see below)* | Independent track. |

Only **GHCR** is used as a distribution channel. There are no Docker Hub,
Quay, or mirror publications at this time.

The `server/` overlay is treated as a **separate publishable artifact**
with its own release track. Its versioning and cadence may diverge from
the base image; consumers must not assume `server/vX.Y.Z` matches
`base/vX.Y.Z`. Until a dedicated server publishing workflow is in place,
treat any `server/` tags as previews; this document will be updated when
the server pipeline is enabled.

## Versioning

Semantic Versioning ([semver.org](https://semver.org)) is applied to the
**runtime contract** of the base image, not to internal implementation
details. The runtime contract is defined in
[base/README.md](base/README.md) and includes:

- Environment variables (names, defaults, semantics).
- Lifecycle hook phases and their execution order.
- On-disk paths and writability expectations.
- Supported database drivers.
- Default behaviours (e.g. JSON:API read-only by default).

Mapping:

| Change                                                    | Bump  |
| --------------------------------------------------------- | ----- |
| Backwards-incompatible runtime contract change.           | MAJOR |
| Additive runtime contract change (new env var, new hook). | MINOR |
| Drupal/PHP/Apache patch updates, CVE fixes, internal refactors. | PATCH |

Drupal core minor and patch updates flow through as base-image **patch or
minor** releases unless they break the documented contract.

## Tag policy

Tags are derived automatically by
[.github/workflows/base-image.yml](.github/workflows/base-image.yml) via
`docker/metadata-action`:

| Trigger                 | Published tags                              | Mutability | Recommended use                 |
| ----------------------- | ------------------------------------------- | ---------- | ------------------------------- |
| GitHub Release `vX.Y.Z` | `X.Y.Z`, `X.Y`, `X`, `latest`               | `X.Y.Z` immutable; others floating | Pin `X.Y.Z` in production. |
| GitHub Release `vX.Y.Z-rcN` | `X.Y.Z-rcN`                             | Immutable  | Pre-production validation.      |
| Push to `main`          | `edge`                                      | Floating   | Integration / smoke only.       |
| Pull request            | `pr-<NNN>` (built, **not** pushed)          | n/a        | Internal CI verification.       |

**Production guidance**: pin to an immutable `X.Y.Z` tag and update via
explicit version bumps. Avoid `latest`, `X.Y`, `X`, and `edge` in
production.

Each published tag includes:

- The OCI image (multi-arch: `linux/amd64`, `linux/arm64`).
- An SBOM (generated by BuildKit).
- A SLSA build provenance attestation (`provenance: mode=max`).
- OCI labels: `org.opencontainers.image.{title,description,source,licenses,version,revision,created}`.

Trivy (`CRITICAL`, `HIGH`, `ignore-unfixed`) gates publish: a failing
scan blocks the release.

## Release cadence

### Scheduled monthly patch releases

- A patch release is cut on the **first Tuesday** of each month.
- Scope: latest Drupal core, PHP, Apache, and base-OS package updates;
  fixes for issues resolved on `main` since the last release.
- A scheduled release may be skipped only if `main` has had **no
  functional changes** since the previous release; the skip is recorded
  in [CHANGELOG.md](CHANGELOG.md) under that month's `Unreleased` →
  released-as-empty entry.

### Emergency security releases

- An off-cycle release is cut **as soon as practical** when a relevant
  CVE is disclosed in any of:
  - Drupal core or a bundled module.
  - PHP or a bundled extension.
  - Apache `httpd`.
  - The upstream PHP base image (`php:8.3-apache`) or its OS packages.
- Emergency releases are normal `vX.Y.Z` patch releases on the affected
  minor lines. The changelog entry is tagged `### Security` and references
  the CVE identifier(s) and the GitHub Security Advisory once published.
- If a fix requires a runtime contract change, the release is a `MINOR`
  bump (or `MAJOR` if breaking) rather than a patch, and the breaking
  change is called out at the top of [base/CHANGELOG.md](base/CHANGELOG.md).

### Pre-releases

Release candidates (`vX.Y.Z-rcN`) are optional and used when a release
introduces non-trivial contract changes that benefit from external
validation before floating tags move.

## Support window

- **Latest minor (`X.Y`)**: receives all monthly patch releases and all
  emergency security releases.
- **Previous minor (`X.(Y-1)`)**: receives emergency security releases
  for **90 days** after the next minor is published, then is end-of-life.
- **Older minors**: not supported. Consumers should upgrade.
- **Major lines**: when a new major (`(X+1).0.0`) is published, the
  previous major receives security-only patches for **180 days**.

End-of-life dates are recorded in [base/CHANGELOG.md](base/CHANGELOG.md)
on the release that triggers them.

## Upgrading

1. Read the relevant entries in [base/CHANGELOG.md](base/CHANGELOG.md)
   between your current and target version. Pay particular attention to
   `### Changed` and `### Removed` for runtime contract impact.
2. Update the `BASE_IMAGE` ARG in your downstream Dockerfile to the new
   immutable `X.Y.Z` tag.
3. Rebuild and run smoke tests against your downstream overlay.
4. For MAJOR upgrades, expect contract changes; consult the release
   notes for migration steps.
5. For emergency security releases, prefer an in-place patch upgrade
   (same minor) and redeploy promptly.

## Release procedure (maintainers)

1. Confirm `main` is green (CI build, smoke, Trivy scan).
2. Move the relevant `[Unreleased]` entries in
   [base/CHANGELOG.md](base/CHANGELOG.md) and [CHANGELOG.md](CHANGELOG.md)
   under a new `[X.Y.Z] - YYYY-MM-DD` heading.
3. Open a release PR; merge after review.
4. Tag the merge commit:
   ```bash
   git tag -a vX.Y.Z -m "drx-drupal-base vX.Y.Z"
   git push origin vX.Y.Z
   ```
5. Publish a GitHub Release for the tag. The `release: published` event
   triggers [.github/workflows/base-image.yml](.github/workflows/base-image.yml),
   which builds, scans, signs, and pushes all derived tags.
6. Verify the published tags resolve on GHCR and the SBOM/provenance
   attestations are attached.
7. For emergency security releases, publish the corresponding GitHub
   Security Advisory once the release is live.
